## Core Concept: KL Regularization

KL penalties discourage the policy from drifting too far from a trusted reference model:

```
J = E_pi[r] - beta * KL(pi || pi_ref)
```

This balances **reward maximization** with **safety constraints**.

### Symbol Glossary

- **pi**: Current policy distribution over actions
- **pi_ref**: Reference policy distribution (trusted baseline)
- **r**: Reward per action
- **beta**: KL penalty strength (higher = more conservative)
- **J**: Overall objective to maximize

### Why This Matters

<Note>
  Without regularization, reinforcement learning can lead to reward hacking where models find unexpected ways to maximize rewards that violate safety constraints or designer intent.
</Note>

<Accordion title="Intuition: Safety Through Anchoring">
  Imagine training a chatbot with reinforcement learning:
  
  - **Without KL penalty**: Model might learn to generate responses that hack the reward model (e.g., extremely long repetitive text that exploits scoring bugs)
  
  - **With KL penalty**: Model is constrained to stay close to the reference policy (e.g., a supervised-trained base model), only making gradual improvements
  
  The penalty ensures changes are incremental and reviewable, not chaotic.
</Accordion>

## Derivation Steps

<Steps>
  <Step title="Compute Expected Reward">
    Calculate the expected reward under the current policy:
    ```
    E_pi[r] = sum(pi(a) * r(a) for all actions a)
    ```
  </Step>
  
  <Step title="Compute KL Divergence">
    Measure how much the policy has shifted from reference:
    ```
    KL(pi || pi_ref) = sum(pi(a) * log(pi(a) / pi_ref(a)))
    ```
  </Step>
  
  <Step title="Combine with Penalty">
    Subtract the weighted KL penalty from the reward objective:
    ```
    J = E_pi[r] - beta * KL(pi || pi_ref)
    ```
  </Step>
</Steps>

### Example Implementation

<Accordion title="Reference Solution">
  ```python
  def kl_regularized_objective(
      policy_probabilities: list[float],
      reference_probabilities: list[float],
      rewards: list[float],
      regularization_beta: float,
  ) -> float:
      expected_reward = sum(
          probability * reward
          for probability, reward in zip(policy_probabilities, rewards)
      )
      kl_divergence = sum(
          probability * math.log(probability / reference_probability)
          for probability, reference_probability in zip(
              policy_probabilities,
              reference_probabilities,
          )
          if probability > 0.0
      )
      return expected_reward - regularization_beta * kl_divergence
  ```
  
  **Key insights:**
  - Expected reward is a weighted sum over actions
  - KL divergence uses natural logarithm
  - Skip zero-probability actions to avoid log(0)
  - Beta controls the safety-performance tradeoff
</Accordion>

### Toy Example

```python
policy    = [0.6, 0.3, 0.1]
reference = [0.5, 0.4, 0.1]
rewards   = [1.0, 0.2, -0.3]
beta      = 0.1

# Expected reward:
# E[r] = 0.6*1.0 + 0.3*0.2 + 0.1*(-0.3) = 0.63

# KL divergence:
# KL = 0.6*log(0.6/0.5) + 0.3*log(0.3/0.4) + 0.1*log(0.1/0.1)
#    ≈ 0.6*0.182 + 0.3*(-0.288) + 0.1*0
#    ≈ 0.0228

# Objective:
# J = 0.63 - 0.1*0.0228 ≈ 0.6277
```

The objective decreases slightly after the KL penalty, encouraging conservative policy updates.

<Warning>
  **Numerical Stability**: When `probability = 0`, skip the KL term to avoid computing `log(0)`. The limit as p→0 of p*log(p) is 0.
</Warning>

## The Role of Beta

<Accordion title="Tuning the Safety-Performance Tradeoff">
  The `beta` parameter controls how conservative the policy updates are:
  
  - **beta = 0**: No safety constraint, pure reward maximization (dangerous)
  - **beta → ∞**: Policy frozen at reference (too conservative)
  - **beta ≈ 0.01-0.1**: Typical range for RLHF in language models
  
  Larger beta means:
  - Slower learning
  - Safer exploration
  - Less reward hacking
  - More stable training
</Accordion>
