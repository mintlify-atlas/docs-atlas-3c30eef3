## Core Concept: Vision Transformers

Vision transformers process images as a sequence of patch tokens instead of pixels:

```
patch_count = (height / patch_size) * (width / patch_size)
```

Each patch becomes a **token** fed into a transformer encoder, just like words in NLP.

### Symbol Glossary

- **height, width**: Image dimensions (pixels)
- **patch_size**: Square patch edge length (e.g., 16x16)
- **patch_count**: Number of patch tokens
- **patch_vector**: Flattened patch values (patch_size²)

### Why Patches Instead of Pixels?

<Note>
  Processing individual pixels with attention would be computationally prohibitive:
  - A 224x224 image has 50,176 pixels
  - Attention complexity is O(n²), so pixel-level attention would require 2.5B operations!
  - With 16x16 patches, we get only 196 tokens (78,000x fewer operations)
</Note>

## How ViT Tokenization Works

<Steps>
  <Step title="Split into Patches">
    Divide the image grid into non-overlapping square windows of size `patch_size × patch_size`
  </Step>
  
  <Step title="Flatten Each Patch">
    Convert each 2D patch into a 1D vector by concatenating rows
    ```
    [a b]  →  [a, b, c, d]
    [c d]
    ```
  </Step>
  
  <Step title="Feed to Transformer">
    Treat the sequence of flattened patches as tokens and process with transformer encoder
  </Step>
</Steps>

<Accordion title="Intuition: Patches as Visual Words">
  Just as language models process **words** (not individual characters), vision transformers process **patches** (not individual pixels).
  
  - Each patch captures local visual patterns
  - Transformer attention learns relationships between patches
  - Position embeddings preserve spatial structure
  - A [CLS] token aggregates global information
  
  This allows ViTs to learn both local features and global context efficiently.
</Accordion>

### Requirements

- Input: 2D image as list of lists (square, size divisible by patch_size)
- Output: List of flattened patch vectors
- Each patch should be flattened row-by-row
- Patches should be extracted in row-major order (left-to-right, top-to-bottom)

### Example Implementation

<Accordion title="Reference Solution">
  ```python
  def patchify_image(
      image: list[list[float]],
      patch_size: int,
  ) -> list[list[float]]:
      image_size = len(image)
      patches: list[list[float]] = []
      
      for row_start in range(0, image_size, patch_size):
          for column_start in range(0, image_size, patch_size):
              patch: list[float] = []
              for row_offset in range(patch_size):
                  patch.extend(
                      image[row_start + row_offset][column_start : column_start + patch_size]
                  )
              patches.append(patch)
      
      return patches
  ```
  
  **Key insights:**
  - Outer loops iterate over patch starting positions (step by `patch_size`)
  - Inner loop builds each patch by extracting row slices
  - `extend()` flattens each row into the patch vector
  - Result is a list of patch_count patches, each with patch_size² elements
</Accordion>

### Toy Example

```python
# 4x4 image, patch_size = 2
image = [
    [1, 2, 3, 4],
    [5, 6, 7, 8],
```

## From Patches to Embeddings

<Accordion title="Complete ViT Pipeline">
  After patchification, a full Vision Transformer does:
  
  1. **Linear projection**: Map each patch vector to embedding dimension
     ```
     patch_embedding = patch_vector @ projection_matrix
     ```
  
  2. **Add position embeddings**: Encode spatial location
     ```
     embedded_patch += position_embedding[i]
     ```
  
  3. **Prepend [CLS] token**: Special token for classification
     ```
     sequence = [CLS_token] + patch_embeddings
     ```
  
  4. **Transformer encoder**: Multi-head attention + feedforward
     ```
     output = TransformerEncoder(sequence)
     ```
  
  5. **Classification head**: Linear layer on [CLS] token
     ```
     logits = output[0] @ classification_head
     ```
</Accordion>

## Design Trade-offs

<Accordion title="Choosing Patch Size">
  **Larger patches (e.g., 32x32)**:
  - ✅ Fewer tokens → less computation
  - ✅ Larger receptive field per token
  - ❌ Loss of fine-grained detail
  - ❌ May miss small objects
  
  **Smaller patches (e.g., 8x8)**:
  - ✅ More detailed representation
  - ✅ Better for small objects
  - ❌ More tokens → quadratic attention cost
  - ❌ Requires more training data
  
  **Common choices**:
  - ViT-B/16: 16×16 patches (standard)
  - ViT-L/14: 14×14 patches (better quality)
  - ViT-H/14: 14×14 patches (largest models)
</Accordion>

## Extension: Handling RGB Images

<Note>
  For color images with RGB channels, each patch would have `patch_size² × 3` values:
  
  ```python
  # Flatten all channels together
  patch = []
  for row_offset in range(patch_size):
      for col_offset in range(patch_size):
          for channel in [R, G, B]:
              patch.append(image[row][col][channel])
  ```
  
  A 16×16 RGB patch becomes a 768-dimensional vector.
</Note>
