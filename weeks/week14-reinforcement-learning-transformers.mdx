## Core Concept: Discounted Returns

Discounted return compresses future rewards into one scalar value per timestep:

```
G_t = r_t + gamma * G_{t+1}
```

This is the **total expected reward** from time t onward, with future rewards discounted.

### Symbol Glossary

- **r_t**: Reward at timestep t
- **gamma** (γ): Discount factor (0 < γ ≤ 1)
- **G_t**: Return-to-go from timestep t
- **T**: Episode horizon (final timestep)

### Why Discounting?

<Note>
  Discounting serves multiple purposes:
  - **Mathematical**: Ensures infinite sums converge
  - **Practical**: Rewards soon are more certain than distant future rewards
  - **Behavioral**: Captures preference for immediate rewards (time preference)
  - **Computational**: Helps gradient flow in long sequences
</Note>

## Derivation Steps

<Steps>
  <Step title="Initialize from Terminal State">
    Start at the last timestep:
    ```
    G_T = r_T
    ```
  </Step>
  
  <Step title="Move Backward Through Time">
    Recursively compute returns in reverse:
    ```
    G_t = r_t + gamma * G_{t+1}
    ```
  </Step>
  
  <Step title="Build Return Sequence">
    Collect all G_t values to get return-to-go for every timestep
  </Step>
</Steps>

<Accordion title="Intuition: Future Rewards with Decay">
  Imagine an agent playing a game:
  
  **Trajectory**: [pickup key → open door → find treasure]
  **Rewards**: [0, 0, +100]
  **Discount**: γ = 0.9
  
  Returns (backward calculation):
  - `G_3 = 100` (treasure)
  - `G_2 = 0 + 0.9*100 = 90` (door opens, treasure is 1 step away)
  - `G_1 = 0 + 0.9*90 = 81` (key pickup, treasure is 2 steps away)
  
  Early actions get credit for enabling future rewards, but discounted by distance.
</Accordion>

### Example Implementation

<Accordion title="Reference Solution">
  ```python
  def compute_discounted_returns(
      rewards: list[float],
      discount_factor: float,
  ) -> list[float]:
      running_return = 0.0
      returns: list[float] = []
      
      for reward in reversed(rewards):
          running_return = reward + discount_factor * running_return
          returns.insert(0, running_return)
      
      return returns
  ```
  
  **Key insights:**
  - Iterate through rewards in **reverse** order
  - Maintain a `running_return` that accumulates from the end
  - Insert at position 0 to build the list forward
  - Each step applies the recursive formula: `G_t = r_t + γ * G_{t+1}`
</Accordion>

### Toy Example

```python
rewards = [1.0, 0.0, -0.5, 2.0]
gamma = 0.9

returns = compute_discounted_returns(rewards, gamma)

# Backward calculation:
# G_4 = 2.0
# G_3 = -0.5 + 0.9*2.0 = 1.3
# G_2 = 0.0 + 0.9*1.3 = 1.17
# G_1 = 1.0 + 0.9*1.17 = 2.053

# Expected output:
# [2.053, 1.17, 1.3, 2.0]
```

Early steps include **long-horizon credit** from the final +2.0 reward.

<Warning>
  **Alternative Implementation**: You could also build the list in reverse and then reverse it at the end:
  ```python
  for reward in reversed(rewards):
      running_return = reward + discount_factor * running_return
      returns.append(running_return)
  return list(reversed(returns))
  ```
  Both approaches are correct—choose based on readability preference.
</Warning>

## Connection to Transformers

<Accordion title="Decision Transformer Architecture">
  **Traditional RL**: Learn value function, then derive policy
  
  **Decision Transformer**: Directly model optimal trajectories as sequence modeling
  
  **Input sequence**:
  ```
  [R_1, s_1, a_1, R_2, s_2, a_2, ..., R_t, s_t, a_t]
  ```
  
  Where:
  - `R_t` = return-to-go (computed using this week's function!)
  - `s_t` = state
  - `a_t` = action
  
  **Key insight**: By conditioning on desired return-to-go, the transformer learns to generate actions that achieve that return. No value function needed!
  
  At test time, you can specify high return-to-go to get optimal behavior.
</Accordion>

## Understanding Gamma (γ)

<Accordion title="The Role of the Discount Factor">
  **γ = 1.0** (no discounting):
  - All future rewards weighted equally
  - Can cause instability in infinite-horizon tasks
  - Appropriate for episodic tasks with clear endpoints
  
  **γ = 0.99** (typical):
  - Balances immediate and future rewards
  - Horizon of ~100 steps (1/0.01)
  - Standard for many deep RL tasks
  
  **γ = 0.9** (more myopic):
  - Stronger preference for immediate rewards
  - Horizon of ~10 steps
  - Useful for sparse reward environments
  
  **γ = 0.0** (greedy):
  - Only immediate rewards matter
  - Degenerates to one-step optimization
  - Rarely useful in practice
</Accordion>

## Real-World Applications

- **Decision Transformer**: Offline RL via sequence modeling
- **Trajectory Transformer**: Dynamics modeling + planning
- **Policy Gradient**: Returns as targets for value estimation
- **Advantage estimation**: GAE uses returns for variance reduction
- **Monte Carlo tree search**: Return-based action selection

## Beyond Basic Returns

<Note>
  **Advanced return estimators**:
  
  - **GAE (Generalized Advantage Estimation)**: Blends n-step returns for bias-variance tradeoff
  - **TD(λ)**: Exponentially weighted mixture of n-step returns
  - **V-trace**: Off-policy return correction
  - **Retrace**: Another off-policy return estimator
  
  All build on the fundamental discounted return formula from this week.
</Note>
